# Authoring loop — Playwright agent CLI (`@playwright/cli`)

One of two interchangeable authoring loops for the E2E module (the other is `../authoring-mcp/`).
This one drives test authoring through the **Playwright agent CLI** (`playwright-cli`) — a
shell-native loop instead of the MCP's structured-tool loop. Pick **either** this or the MCP, not
both.

See the Playwright agent-CLI docs: <https://playwright.dev/agent-cli/introduction>.

## Files here

| File | Role |
|---|---|
| `package.json` | The CLI-flavored suite package — adds `@playwright/cli` plus the `e2e:auth` / `e2e:report` / `pw:install` scripts on top of the shared `@playwright/test` + `dotenv`. |

That's it — this loop has **no** `.mcp.json` and **no** subagents. Its analog to the MCP's
`init-agents` is a generator command (below) that installs local reference "skill" docs.

## Why not vendor the skill docs?

The MCP approach ships **modified** agents (we edit their output paths), so those files are the
deliverable. The CLI approach uses the skill docs **unmodified** — they're stock upstream
reference material generated by `playwright-cli install --skills`. Vendoring 11 upstream docs that
drift with each `@playwright/cli` release would just go stale, so we run the generator at adoption
time instead (you already run `npm install` + `playwright install`, so this needs no extra setup
you weren't doing).

## Adoption

From the workspace folder, after copying the shared files (`../playwright.config.ts`,
`../.env.e2e.example`, `../e2e/`, `../gitignore-additions.txt` — see `../README.md`):

1. Copy this directory's `package.json` into the workspace root.
2. Install: `npm install` (or bun/pnpm) then `npm run pw:install` (browsers).
3. **Install the CLI skills** at the workspace level:
   ```bash
   npx playwright-cli install --skills            # default target: claude
   ```
   This writes reference docs to `.claude/skills/playwright-cli/` (`SKILL.md` + `references/*.md`,
   notably `spec-driven-testing.md`, `storage-state.md`, `test-generation.md`, `tracing.md`,
   `video-recording.md`). **Commit them** — they're auto-discovered by Claude Code from
   `.claude/skills/`, so committing makes the agent setup reproducible. (If `playwright-cli` isn't
   global, use `npx playwright-cli …` everywhere; `npm install -g @playwright/cli@latest` installs
   it globally.)
4. Document the loop in the workspace `CLAUDE.md` — see the snippet below.

## The authoring loop (document this in `CLAUDE.md`)

To cover a spec, Claude drives `playwright-cli` against the **running app** to discover and verify
selectors against the real DOM, reusing a role session, then writes the test:

```
- **Authoring loop (agent CLI, NOT the MCP):** author tests with the Playwright agent CLI.
  Skills are installed at `.claude/skills/playwright-cli/` (auto-discovered) — drive the live app
  with `playwright-cli` (`open` / `snapshot` / `click` / `fill` / `state-save`) to discover and
  verify selectors against the real DOM, reusing a role session, then write `e2e/<slug>.spec.ts`
  (same kebab slug as the spec; spec files stay flat at the `e2e/` root, non-spec helpers live in
  `e2e/support/`), run the suite, and iterate on failures using the trace/video artifacts. Do NOT
  use the Playwright MCP here. If `playwright-cli` isn't global, use `npx playwright-cli`.
```

The triage rule is the same as everywhere in this module: a red test is fixed by the app **code**,
the **test**, or (only on a human call) the **spec** — never by weakening an assertion to match a
bug. There is no separate "healer" subagent here; the guardrail lives in the workflow, and the
skill docs' `spec-driven-testing.md` covers the plan/generate/heal cycle via the CLI.
